# Web3 Integration Design Spec

**Date:** 2026-06-18
**Status:** Approved for implementation

---

## What we are building

Two independent Web3 sub-systems added to the existing Steward backend:

1. **Crypto donations** — The Giving Block settlement CSV import as a new donation source (`'crypto'`), plugging into the existing M3 import pipeline with zero changes to core logic
2. **Blockchain audit anchoring** — On period close, a SHA-256 hash of the period's financial data is submitted as calldata to Polygon (or Base), creating a tamper-evident public record verifiable by anyone on a block explorer

---

## Scope

**Included:**
- New `ImportSource` value `'crypto'`
- Migration adding `'crypto'` to source check constraints
- Column map for The Giving Block settlement CSV
- `chain_anchors` table + RLS
- `anchor_tx_hash` column on `reconciliation_periods`
- `anchorPeriod` internal helper — called by `closePeriod`
- `reanchorPeriod` server action — admin retry for failed anchors
- `getAnchor` server action — read anchor for a period
- `/api/verify/[orgId]/[year]/[month]` route — public verification endpoint
- `viem` integration with platform wallet

**Excluded:**
- Direct crypto wallet management for churches
- Smart contracts (calldata-only approach)
- NFT receipts, token-gating, DAO governance
- Gift Aid on crypto donations (HMRC does not permit it)
- UI for anchor verification (backend + API only)

```
Backend Impact Check:
- Business rule:
    Crypto donations: import settlement CSV from The Giving Block;
    amount_pence = Settlement Amount (GBP) × 100; source_reference = Transaction ID.
    Crypto donations are never Gift Aid eligible regardless of donor flag.

    getAnchor: any org member
    /api/verify: public (no auth required)
- RLS affected: chain_anchors (new table)
- Audit log required:
    chain_anchor.created — on successful anchor
    chain_anchor.failed — on anchor failure
    chain_anchor.reanchored — on manual retry
- Stripe/payment impact: none
- Migration required: yes — migration 018
- Tests required:
    Zod schema tests, column map pure function tests,
    buildAnchorPayload determinism tests, SHA-256 hash tests
- Risk level: MEDIUM — anchor failure must not block period close;
    private key exposure would be catastrophic (server-side only)
```

### Column map (`src/lib/importMaps/theGivingBlock.ts`)

The Giving Block settlement CSV columns:

| CSV column | Maps to | Notes |
|---|---|---|
| `Settlement Date` | `transaction_date` | YYYY-MM-DD |
| `Settlement Amount (GBP)` | `amount_pence` | × 100, integer |
| `Transaction ID` | `source_reference` | unique per donation |
| `Donor Name` | `notes` | no donor matching at import |
| `Cryptocurrency`, `Crypto Amount`, `Exchange Rate` | `raw_row` jsonb only | preserved, not modelled |

Gift Aid: crypto donations cannot be Gift Aid eligible — HMRC does not permit Gift Aid on cryptocurrency donations. This is enforced by the donor `gift_aid_eligible` flag and `isDeclarationActive` logic already in M5 — no additional enforcement needed at import time. The `raw_row` preserves the original crypto details for audit.

## Sub-system 2: Blockchain audit anchoring

### Migration 018 (continued)

```sql
-- Add anchor_tx_hash to reconciliation_periods
alter table reconciliation_periods
  add column anchor_tx_hash text;

-- chain_anchors table
create table if not exists chain_anchors (
  id uuid primary key default gen_random_uuid(),
  organisation_id uuid not null references organisations(id) on delete cascade,
  period_id uuid not null references reconciliation_periods(id),
  chain text not null check (chain in ('polygon', 'base')),
  chain_id integer not null,
  tx_hash text not null,
  block_number bigint not null,
  anchor_hash text not null,
  anchor_data jsonb not null,
  prev_anchor_tx_hash text,

  anchored_by uuid references auth.users(id)
);

alter table chain_anchors enable row level security;

create policy "org members can view anchors"
on chain_anchors for select
using (public.is_org_member(organisation_id));

-- No INSERT/UPDATE/DELETE RLS — admin client only

create index if not exists idx_chain_anchors_org on chain_anchors(organisation_id);
create index if not exists idx_chain_anchors_period on chain_anchors(period_id);
```

### Canonical anchor payload

Deterministic JSON — keys in fixed order, donation_ids and audit_log_ids sorted alphabetically:

```ts
interface AnchorPayload {
  steward_version: '1';
  organisation_id: string;
  period_id: string;
  year: number;
  month: number;
  total_pence: number;
  donation_count: number;
  donation_ids: string[];      // sorted ascending
  audit_log_ids: string[];     // sorted ascending, filtered to this period
  closed_by: string;
  closed_at: string;           // ISO timestamp
  prev_anchor_tx_hash: string | null;
}
```

Hash = `SHA-256(JSON.stringify(payload))` — hex string.

### Anchoring flow (inside `closePeriod`, after successful period write)

```
1. Fetch donation IDs for period (sorted)
2. Fetch audit_log IDs for period (sorted)
3. Build AnchorPayload, JSON.stringify, SHA-256 hash
4. Submit zero-value tx to Polygon burn address (0x000...dEaD)
   with hash as hex calldata via viem
5. Wait for tx receipt (1 block confirmation)
6. Insert chain_anchors row
7. Update reconciliation_periods.anchor_tx_hash
8. Write audit log chain_anchor.created
```

If step 4–8 fails:
- Period remains closed (financial data is correct)
- Write audit log `chain_anchor.failed` with error message
- Return `{ data: period }` — period close success is not rolled back
- `reanchorPeriod` can retry later

### Infrastructure (`src/lib/blockchain/anchor.ts`)

Pure module — no Supabase imports. Exports:

```ts
export function buildAnchorPayload(params: {...}): AnchorPayload
export function hashAnchorPayload(payload: AnchorPayload): string  // hex SHA-256
export async function submitAnchorTx(hash: string): Promise<{ tx_hash: string; block_number: bigint }>
```

Environment variables (server-side only, never client):
- `ANCHOR_CHAIN` — `'polygon'` | `'base'`
- `ANCHOR_CHAIN_ID` — `137` (Polygon) | `8453` (Base)
- `ANCHOR_WALLET_PRIVATE_KEY` — platform wallet private key
- `ANCHOR_RPC_URL` — Alchemy/QuickNode RPC endpoint
- `ANCHOR_BURN_ADDRESS` — `0x000000000000000000000000000000000000dEaD`

### Server actions (`src/app/actions/v2/chainAnchors.ts`)

```
anchorPeriod(period_id, organisation_id, actorId): Promise<ActionResult<ChainAnchor>>
  → internal only (not exported as server action)
  → called by closePeriod after successful period write
  → builds payload, hashes, submits tx, stores anchor

reanchorPeriod(input: ReanchorInput): Promise<ActionResult<ChainAnchor>>
  → requireRole(owner/admin/finance_manager)
  → checks no existing successful anchor for period
  → same flow as anchorPeriod
  → writes audit log chain_anchor.reanchored

getAnchor(input: GetAnchorInput): Promise<ActionResult<ChainAnchor | null>>
  → requireRole(any member)
  → returns anchor for period or null
```

### Public verification route (`src/app/api/verify/[orgId]/[year]/[month]/route.ts`)

No auth required — public endpoint.

```
GET /api/verify/:orgId/:year/:month

Response:
{
  verified: boolean,
  anchor_hash: string,        // what Steward has
  tx_hash: string,            // Polygonscan link
  block_number: number,
  chain: string,
  chain_id: number,
  anchored_at: string,
  polygonscan_url: string     // https://polygonscan.com/tx/0x...
}
```

Verification logic:
1. Load `chain_anchors` row for period (admin client — no RLS bypass needed for public data)
2. Re-hash `anchor_data` jsonb → compare to stored `anchor_hash`
3. Optionally fetch tx calldata from chain via RPC and compare (expensive — skip for MVP, document as manual step)
4. Return result

### Tests (`src/__tests__/v2/web3.test.ts`)

- `theGivingBlock` column map: correct field mapping, amount × 100 integer, source_reference extraction, raw_row preservation
- `buildAnchorPayload`: deterministic output (same input → same JSON), donation_ids sorted, audit_log_ids sorted
- `hashAnchorPayload`: known input → known SHA-256 output (snapshot test)
- `reanchorPeriodSchema`: valid, rejects non-uuid
- `getAnchorSchema`: valid

### Security

- `ANCHOR_WALLET_PRIVATE_KEY` server-side only — never in any client bundle, never logged
- Platform wallet holds only enough MATIC for ~6 months of gas — minimise exposure
- Anchor failure does not block period close — financial integrity preserved
- `chain_anchors` has no client-facing INSERT/UPDATE/DELETE RLS
- Public verification endpoint reads only `chain_anchors` (no sensitive org data exposed)
- `anchor_data` jsonb contains only IDs and aggregates — no PII, no amounts per donor

- [ ] `reconciliation_periods.anchor_tx_hash` updated
- [ ] Anchor failure does not block period close
- [ ] `reanchorPeriod` retries a failed anchor
- [ ] `GET /api/verify/:orgId/:year/:month` returns verified=true for anchored period
- [ ] `buildAnchorPayload` is deterministic (same input → same output always)
- [ ] Platform wallet private key never appears in client bundle or logs
